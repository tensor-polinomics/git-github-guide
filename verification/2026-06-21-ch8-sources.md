## D1. GitHub file-size limits

Access date: 2026-06-21
Source: GitHub Docs, "About large files on GitHub"
URL: https://docs.github.com/en/repositories/working-with-files/managing-large-files/about-large-files-on-github
Used in: Ch.8 (why data does not belong in Git)

Verified claims: a file larger than 50 MiB triggers a Git warning but
still pushes; GitHub blocks files larger than 100 MiB (must use Git
LFS beyond that); a file added through the browser can be no larger
than 25 MiB.

## D2. Git LFS: what it does + GitHub free quota

Access date: 2026-06-21
Source: GitHub Docs, "About Git Large File Storage" and "About billing
for Git Large File Storage"
URL: https://docs.github.com/en/repositories/working-with-files/managing-large-files/about-git-large-file-storage
      https://docs.github.com/billing/managing-billing-for-git-large-file-storage/about-billing-for-git-large-file-storage
Used in: Ch.8 (documented-not-executed; proxy blocks git-lfs install)

Verified claims: Git LFS replaces large files in the repository with
small pointer files, while the actual file content is stored on a
separate LFS store and still versioned. GitHub's free tier includes
1 GB of LFS storage and 1 GB/month of bandwidth; maximum LFS file size
is 2 GB on GitHub Free and Pro. Billing beyond the free quota is
metered (storage by usage, bandwidth per GiB downloaded).

Notes: exact pricing model has shifted (data packs -> metered) and may
change again; the book states only the free-tier 1 GB / 1 GB figures
and the "pointers, not contents" mechanism, both stable.

## D3. DVC (Data Version Control): mechanism

Access date: 2026-06-21
Source: DVC Docs, "Remote Storage" and "Get Started"
URL: https://dvc.org/doc/user-guide/data-management/remote-storage
      https://dvc.org/doc/start
Used in: Ch.8 (documented-not-executed)

Verified claims: DVC replaces large files, directories, and models with
small metafiles (`.dvc` files / pointers) that are committed to Git,
while the data itself lives in a DVC cache and is pushed to external
remote storage (S3, Google Drive, Azure, SSH, etc.). `dvc push` /
`dvc pull` move data to and from the remote; `dvc checkout` syncs the
workspace to the version named by the current `.dvc` files. Git tracks
the small pointer, not the bytes.

## D4. Zenodo: DOIs, size limits, GitHub integration

Access date: 2026-06-21
Source: Zenodo Support / Help, "What are the size limitations of
Zenodo?" and "Manage files"; Zenodo + GitHub guide
URL: https://support.zenodo.org/help/en-gb/1-upload-deposit/80-what-are-the-size-limitations-of-zenodo
      https://help.zenodo.org/docs/deposit/manage-files/
Used in: Ch.8 and Ch.15 (documented-not-executed)

Verified claims: Zenodo accepts up to 50 GB per record with a maximum
of 100 files by default, with a one-time quota increase up to 200 GB
available on request. Zenodo mints a DOI for each record. Zenodo is
integrated with GitHub: it can automatically archive a tagged GitHub
release and assign that archive its own DOI.

## D6. renv: reproducible R environments

Access date: 2026-06-21
Source: renv docs, "Introduction to renv"
URL: https://rstudio.github.io/renv/articles/renv.html
Used in: Ch.8 (documented-not-executed; R not installed in sandbox)

Verified claims: renv gives each R project a private library and
records exact package versions in a lockfile named `renv.lock`.
`renv::snapshot()` writes the lockfile from the project's current
packages; `renv::restore()` reinstalls exactly those versions from the
lockfile. To reproduce an environment, commit `renv.lock` (along with
`.Rprofile` and `renv/activate.R`) to version control. This is the R
analogue of the book's `uv.lock` workflow.

## D7. conda: environment.yml is not a lock; conda-lock is

Access date: 2026-06-21
Source: conda Docs, "Managing environments"; conda-lock project
URL: https://docs.conda.io/projects/conda/en/stable/user-guide/tasks/manage-environments.html
      https://github.com/conda/conda-lock
Used in: Ch.8 (documented-not-executed)

Verified claims: `conda env export` to an `environment.yml` does not
guarantee a fully pinned dependency tree and is platform-specific, so
it is not a true lockfile. conda-lock performs a conda solve per target
platform and produces a fully pinned lockfile (down to the build
string, e.g. `pandas=2.2.2=py311ha02d727_0`), giving cross-platform
reproducibility that a plain `environment.yml` cannot.

Notes: book uses this as the cautionary contrast: an `environment.yml`
"requirements" file is necessary but not sufficient for reproducibility;
a true lock (uv.lock, renv.lock, conda-lock) is what pins the build.
