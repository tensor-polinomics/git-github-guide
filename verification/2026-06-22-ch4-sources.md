## S1. `init.defaultBranch`: configurable initial branch since Git 2.28

Access date: 2026-06-22
Source: git-scm, `git config` docs (2.28.0) and `git-init` docs;
GitHub "renaming" guidance repo.
URL: https://git-scm.com/docs/git-config/2.28.0
      https://git-scm.com/docs/git-init
      https://github.com/github/renaming
Used in: Ch.4 (sensible defaults), Ch.2/Ch.3 (why default branch is
`main`).

Verified claims: Git 2.28 (released 2020) introduced the
`init.defaultBranch` configuration variable. From 2.28 onward,
`git init` names the initial branch after `init.defaultBranch`; if
that value is unset, it defaults to `master` and `git init` prints a
hint advising the user to set `init.defaultBranch`. Setting
`git config --global init.defaultBranch main` makes `main` the initial
branch for all new repositories and suppresses the hint. Confirmed
live in-sandbox on Git 2.34.1 (transcripts/04_defaultbranch.txt): the
exact hint text appears before the variable is set, and
`symbolic-ref --short HEAD` reports `main` after.

## S2. SSH connection test message (`ssh -T git@github.com`)

Access date: 2026-06-22
Source: GitHub Docs, "Testing your SSH connection."
URL: https://docs.github.com/en/authentication/connecting-to-github-with-ssh/testing-your-ssh-connection
Used in: Ch.4 (verify the SSH key works).

Verified claims: the command to test an SSH key against GitHub is
`ssh -T git@github.com`. On success GitHub returns exactly:
"Hi USERNAME! You've successfully authenticated, but GitHub does not
provide shell access." The absence of shell access is expected and is
not an error. On first connection ssh prints a host-authenticity
warning showing GitHub's ED25519 key fingerprint, which the user
should verify before typing `yes`. A misconfigured key yields
"Permission denied (publickey)." NOTE: the success banner is captured
on the user's Mac (path A) because it requires the user's real key to
be registered on their real GitHub account; redact the username if
desired.

FILLED 2026-06-22 (transcripts/04_macos_setup.txt): real macOS run on
the user's Mac confirms the banner verbatim (username redacted), and
`git --version` = 2.53.0 (Homebrew). FIELD FINDING folded into Ch.4:
the default `~/.ssh/id_ed25519.pub` is NOT guaranteed to exist; this
Mac uses named keys (id_ed25519_work, id_ed25519_personal), so
`ssh-keygen -lf ~/.ssh/id_ed25519.pub` errors with "No such file."
Ch.4 now teaches `find ~/.ssh -name '*.pub'` first and never assumes
the default filename. The two real public-key SHA256 fingerprints
were further REDACTED in both the public transcript and the chapter
(transcripts/ is public); CLAUDE.md permits showing a public-key
fingerprint, but these are the user's real keys with no pedagogical
need to expose.

## S3. Credential helpers (HTTPS) per OS

Access date: 2026-06-22
Source: GitHub Docs, "Caching your GitHub credentials in Git" and
"Updating credentials from the macOS Keychain"; Pro Git "Credential
Storage."
URL: https://docs.github.com/en/get-started/git-basics/caching-your-github-credentials-in-git
      https://docs.github.com/en/get-started/git-basics/updating-credentials-from-the-macos-keychain
      https://git-scm.com/book/en/v2/Git-Tools-Credential-Storage
Used in: Ch.4 (SSH vs HTTPS sidebar; what to use if you pick HTTPS).

Verified claims: for HTTPS remotes GitHub recommends Git Credential
Manager (GCM) or the GitHub CLI to store credentials. macOS Git ships
an `osxkeychain` helper that caches credentials in the system keychain;
on macOS GCM configures Git automatically. Windows: enable Git
Credential Manager during the Git for Windows install (backs onto the
Windows Credential Store). Linux: install GCM separately, or use the
built-in `cache` (in-memory, time-limited) or `store` (plaintext file,
discouraged) helpers. The book recommends SSH as the primary path and
mentions credential helpers only for the HTTPS alternative; no volatile
numbers are cited.

## S4. Line endings: `core.autocrlf` per OS, and `.gitattributes`

Access date: 2026-06-22
Source: GitHub Docs, "Configuring Git to handle line endings"; Pro Git
"Customizing Git - Git Configuration."
URL: https://docs.github.com/en/get-started/git-basics/configuring-git-to-handle-line-endings
      https://git-scm.com/book/en/v2/Customizing-Git-Git-Configuration
Used in: Ch.4 (sensible defaults; cross-platform collaboration).

Verified claims: on Windows, set `core.autocrlf true` (checks out CRLF,
commits LF). On macOS and Linux, set `core.autocrlf input` (commits LF,
leaves checkout untouched). A repo-level `.gitattributes` with
`* text=auto` overrides per-user `core.autocrlf` for every contributor
and is the durable, project-portable fix; `text eol=crlf` / `eol=lf`
force a specific ending for named files. The book sets
`core.autocrlf input` on the macOS/Linux build machine
(transcripts/04_defaults.txt) and documents `true` for Windows as
not-locally-run.

## S5. Install methods per OS (documented; Windows not run here)

Access date: 2026-06-22
Source: git-scm downloads page; Git for Windows project.
URL: https://git-scm.com/downloads
      https://git-scm.com/download/win
Used in: Ch.4 (install section).

Verified claims: the official cross-platform download entry point is
git-scm.com/downloads. Windows users install "Git for Windows," which
bundles Git Bash (a bash emulation) and optionally Git Credential
Manager; this is the path documented but NOT run in-sandbox (no
Windows host) and is flagged as such in the chapter. macOS: Git is
commonly installed via Homebrew (`brew install git`) or the Xcode
Command Line Tools; the book uses the user's Homebrew Git (path A).
Linux (Debian/Ubuntu): `sudo apt-get install git`; the sandbox already
ships Git 2.34.1 (transcripts/04_version.txt), so the Linux version
output is real. Exact installer wording and bundled-component defaults
drift between releases; re-verify the Windows installer screens against
the current Git for Windows build before any future edition.
